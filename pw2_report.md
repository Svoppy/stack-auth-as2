## Краткое описание репозитория и архитектурного контекста

**Репозиторий:** [stack-auth/stack](https://github.com/stack-auth/stack)
**Лицензия:** MIT / AGPL
**Стек:** Next.js, React, TypeScript, PostgreSQL (Prisma ORM), Redis, Svix (webhooks)

**Stack Auth** — открытая managed-платформа аутентификации и управления пользователями. Основные функциональные блоки:

| Компонент | Описание |
|-----------|----------|
| **Backend API** (`apps/backend`) | REST API для аутентификации, сессий, пользователей, проектов |
| **Dashboard** (`apps/dashboard`) | Административная панель для разработчиков |
| **SDK** (`packages/stack`, `sdks/`) | Клиентские SDK (Next.js, React, Swift, REST) |
| **База данных** | PostgreSQL через Prisma ORM |
| **Email** | Кастомные шаблоны + внешний SMTP / email-провайдер |
| **Webhooks** | Svix — доставка событий в приложения разработчиков |
| **OAuth** | Интеграция с Google, GitHub, Microsoft и другими провайдерами |

Система обеспечивает: OAuth-аутентификацию, парольную аутентификацию, magic links, passkeys, JWT/refresh-token сессии, RBAC, M2M-аутентификацию, multi-tenancy (команды/проекты).

# Задание 1. Построение диаграммы потоков данных (DFD)

## 1.1 Границы анализа системы

Анализ охватывает полный жизненный цикл аутентификации и управления пользователями:
- от получения учётных данных от конечного пользователя
- до выдачи токенов, управления сессиями и доставки событий внешним системам

**Вне анализа:** внутренние CI/CD процессы, мониторинг инфраструктуры, CDN.

---

## 1.2 Элементы DFD

### Внешние сущности (External Entities)

| ID | Название | Описание | Уровень доверия |
|----|----------|----------|-----------------|
| **E1** | Конечный пользователь | Браузер/мобильный клиент, выполняющий вход или регистрацию | Низкий — не доверенный |
| **E2** | Приложение разработчика | Серверное или клиентское приложение, интегрирующее Stack Auth SDK | Средний — доверяется после аутентификации API-ключом |
| **E3** | OAuth-провайдер | Внешние IdP: Google, GitHub, Microsoft, Apple и др. | Средний — доверяется по HTTPS + OIDC |
| **E4** | Email-сервис | Внешний SMTP-сервер или email API (SendGrid, Resend и др.) | Средний — только исходящий трафик |
| **E5** | Webhook-потребитель | Серверный эндпоинт приложения разработчика, принимающий события | Средний — верифицируется HMAC-подписью Svix |

### Процессы (Processes)

| ID | Название | Описание |
|----|----------|----------|
| **P1** | Аутентификация и управление сессиями | Проверка учётных данных, выдача JWT/refresh-токенов, управление сессиями |
| **P2** | Управление OAuth-токенами | Инициация OAuth-флоу, обработка callback, обмен кодов на токены, обновление |
| **P3** | Управление пользователями и RBAC | CRUD пользователей, назначение ролей/разрешений, управление командами |
| **P4** | Диспетчер email-уведомлений | Генерация и отправка писем (верификация, сброс пароля, приветствие) |
| **P5** | Обработчик вебхуков | Генерация событий, подписание payload через Svix, доставка потребителям |
| **P6** | API административной панели | Управление проектами, ключами API, пользователями, настройками через Dashboard |

### Хранилища данных (Data Stores)

| ID | Название | Содержимое | Технология |
|----|----------|------------|------------|
| **D1** | База данных пользователей и сессий | Пользователи, хеши паролей, сессии, токены, проекты, роли, разрешения, OAuth-токены, API-ключи | PostgreSQL (Prisma) |
| **D2** | Кэш сессий и токенов | Кратковременные access-токены, blacklist отозванных токенов | Redis / in-memory |
| **D3** | Хранилище email-шаблонов | HTML-шаблоны писем, настройки отправителя, кастомизация по проекту | PostgreSQL (D1) / файловая система |

### Потоки данных (Data Flows)

| ID | От | До | Тип данных | Протокол/способ | Уровень доверия источника |
|----|----|----|------------|-----------------|--------------------------|
| **F1** | E1 | P1 | Учётные данные (email + пароль / OTP / passkey) | HTTPS POST (REST API) | Низкий |
| **F2** | P1 | E1 | Access token (JWT) + refresh token + session info | HTTPS Response + HttpOnly Cookie | Высокий (система) |
| **F3** | P1 | D1 | Запись сессии, обновление last_login, флаги | SQL (Prisma ORM) | Высокий (внутренний) |
| **F4** | D1 | P1 | Запись пользователя, хеш пароля, статус аккаунта | SQL Query Response | Высокий (внутренний) |
| **F5** | E2 | P6 | API-ключ + управляющий запрос (создание пользователя, получение профиля) | HTTPS + Bearer token | Средний |
| **F6** | P3 | D1 | CRUD пользователей, назначение ролей, создание команд | SQL (Prisma ORM) | Высокий (внутренний) |
| **F7** | P2 | E3 | OAuth Authorization Request (client_id, scope, redirect_uri, state, PKCE) | HTTPS Redirect (302) | Средний |
| **F8** | E3 | P2 | OAuth Callback (code, state) → Token Exchange (access_token, id_token) | HTTPS Callback + backchannel HTTPS | Средний |
| **F9** | P1/P3 | P4 | Триггер email: тип события, user_id, данные шаблона | Внутренний вызов / очередь | Высокий (внутренний) |
| **F10** | P4 | E4 | Email payload (получатель, тема, HTML-body) | SMTP / HTTPS API | Средний |
| **F11** | P5 | E5 | Webhook payload (JSON событие: user.created, session.deleted и др.) + HMAC-подпись | HTTPS POST (Svix) | Высокий (подписан) |
| **F12** | E2 | P6 | Dashboard-запрос (управление проектом, настройки OAuth, email-шаблоны) | HTTPS + API-ключ / cookie-сессия | Средний |

### Границы доверия (Trust Boundaries)

| ID | Название | Описание |
|----|----------|----------|
| **TB1** | Интернет / периметр системы | Разделяет внешних пользователей и приложения разработчиков от внутренних процессов Stack Auth. Пересекается потоками F1, F2, F5, F7, F8, F12 |
| **TB2** | Внутренний периметр / зона данных | Разделяет процессинговые компоненты (P1–P6) от хранилищ данных (D1, D2, D3). Пересекается потоками F3, F4, F6, F9 |
| **TB3** | Граница третьих сторон | Разделяет систему Stack Auth от внешних сервисов (E3, E4, E5). Пересекается потоками F7, F8, F10, F11 |

## 1.6 Критичные пересечения границ доверия

### Пересечение 1: F1 пересекает TB1 (E1 → P1)
**Поток:** Учётные данные пользователя (email + пароль / OTP) поступают из интернета в процесс аутентификации.

**Угрозы:**
- **Брутфорс и credential stuffing** — перебор паролей без rate limiting
- **Инъекция учётных данных** — SQL/NoSQL инъекции в полях логина при некорректной валидации
- **Перехват в transit** — если TLS неправильно настроен (downgrade-атаки)
- **Фишинг** — подмена endpoint, пользователь отправляет данные злоумышленнику

### Пересечение 2: F8 пересекает TB3 (E3 → P2)
**Поток:** OAuth callback от внешнего провайдера (code + state) возвращается в систему.

**Угрозы:**
- **CSRF на OAuth callback** — злоумышленник подменяет state и привязывает свой аккаунт к жертве
- **Перехват authorization code** — открытые redirect URI перенаправляют code злоумышленнику
- **OAuth token leakage** — access_token внешнего провайдера хранится/передаётся небезопасно
- **Подмена провайдера** — атака через вредоносный OIDC-конфигурационный endpoint

### Пересечение 3: F5/F12 пересекают TB1 (E2 → P6)
**Поток:** Запросы от приложения разработчика с API-ключами к административному API.

**Угрозы:**
- **Компрометация API-ключа** — утечка ключа из репозитория, логов, конфигурации
- **Broken Object Level Authorization** — разработчик получает доступ к данным чужих проектов
- **Mass assignment** — изменение привилегированных полей через API без должной валидации
- **Replay-атаки** — повторное использование перехваченных запросов без nonce/timestamp

# Задание 2. Сравнение методов оценки риска

## 2.1 Перечень угроз (на основе DFD)

| ID | Угроза | Актив | Источник угрозы | Уязвимый компонент | Возможные последствия |
|----|--------|-------|-----------------|--------------------|-----------------------|
| **T1** | Brute-force / credential stuffing на endpoint аутентификации | Аккаунты пользователей, сессии | Внешний злоумышленник | P1 (F1, TB1) | Захват аккаунта (ATO), утечка пользовательских данных, репутационный ущерб |
| **T2** | Компрометация/утечка API-ключа разработчика | Все данные проекта (пользователи, настройки) | Инсайдер, ошибочный commit, сканер GitHub | P6, D1 (F5, F12, TB1) | Полный доступ к пользовательской базе проекта, изменение настроек OAuth |
| **T3** | CSRF / перехват OAuth authorization code | Аккаунты пользователей, OAuth-токены внешних сервисов | Внешний злоумышленник (MITM, фишинг) | P2 (F7, F8, TB3) | Привязка чужого OAuth к аккаунту злоумышленника, угон сессии |
| **T4** | SQL-инъекция через пользовательский ввод | База данных D1 (все данные) | Внешний злоумышленник | P1, P3, D1 (F1, F6, TB1/TB2) | Дамп всей БД, обход аутентификации, изменение данных |
| **T5** | Подделка/перехват webhook payload | Целостность событий, данные пользователей E5 | Внешний злоумышленник (MITM) или скомпрометированный E5 | P5 (F11, TB3) | Ложные события в приложении разработчика, несанкционированные действия |
| **T6** | Privilege escalation через некорректную проверку RBAC | Данные всех пользователей, административные функции | Аутентифицированный злоумышленник | P3, D1 (F6, TB2) | Доступ к данным других пользователей/команд, административные действия |

## 2.2 Качественная оценка риска

**Шкала:** Низкий / Средний / Высокий / Критический

| ID | Угроза | Вероятность | Последствия | Качественный риск |
|----|--------|-------------|-------------|-------------------|
| T1 | Brute-force / credential stuffing | Высокая (автоматизированные атаки массовые) | Высокие (ATO, утечка данных) | **Критический** |
| T2 | Компрометация API-ключа | Средняя (частые случаи в GitHub) | Критические (полный доступ к проекту) | **Критический** |
| T3 | CSRF / перехват OAuth code | Средняя (требует взаимодействия жертвы) | Высокие (угон аккаунта) | **Высокий** |
| T4 | SQL-инъекция | Низкая (Prisma ORM защищает, но кастомные запросы) | Критические (дамп всей БД) | **Высокий** |
| T5 | Подделка webhook payload | Низкая (Svix HMAC, но потребитель может не проверять) | Средние (ошибочные действия в приложении) | **Средний** |
| T6 | Privilege escalation (RBAC) | Средняя (сложная логика разрешений) | Высокие (межпроектный доступ) | **Высокий** |

## 2.3 Полуколичественная оценка риска

**Формула:** `Риск = Вероятность (1–5) × Последствия (1–5)`

**Шкала вероятности:**
- 1 — Редко (< 1 раза в 5 лет)
- 2 — Маловероятно (1 раза в 2–5 лет)
- 3 — Возможно (1 раза в год)
- 4 — Вероятно (несколько раз в год)
- 5 — Почти наверняка (постоянно)

**Шкала последствий:**
- 1 — Незначительные (минимальный ущерб)
- 2 — Малые (локальный ущерб)
- 3 — Умеренные (ощутимый ущерб)
- 4 — Существенные (значительный ущерб)
- 5 — Катастрофические (критический ущерб для бизнеса)

| ID | Угроза | Вероятность | Последствия | Риск (P×L) | Приоритет |
|----|--------|:-----------:|:-----------:|:----------:|:---------:|
| T1 | Brute-force / credential stuffing | 5 | 4 | **20** | 1 |
| T2 | Компрометация API-ключа | 4 | 5 | **20** | 1 |
| T3 | CSRF / перехват OAuth code | 3 | 4 | **12** | 3 |

## 2.4 FAIR-lite оценка для наиболее критичной угрозы

**Выбрана угроза T2 — Компрометация/утечка API-ключа разработчика** (Риск = 20, максимальные последствия).

### Обоснование выбора
API-ключи Stack Auth предоставляют полный административный доступ к проекту (чтение/изменение всех пользователей, настроек, токенов). Утечка через публичный репозиторий — один из наиболее распространённых векторов атак (GitGuardian фиксирует миллионы таких инцидентов ежегодно).

### FAIR-lite расчёт

**Шаг 1. Частота реализации угрозы (TEF — Threat Event Frequency)**

| Параметр | Значение |
|----------|----------|
| Вероятность случайного коммита ключа в репозиторий за год | ~35% (оценка по отраслевым данным) |
| Вероятность, что ключ будет обнаружен и использован злоумышленником | ~60% (автосканеры GitHub secrets) |
| Расчётная частота: TEF | ~0,21 инцидента/год (≈1 раз в 5 лет на проект) |
| Для платформы с 1000+ проектами разработчиков | ~210 инцидентов/год |

> Для оценки одного среднестатистического проекта-клиента Stack Auth: **TEF ≈ 0,2 событий/год**

**Шаг 2. Ожидаемый ущерб за один инцидент (Loss Magnitude, LM)**

| Статья ущерба | Сумма (USD) | Обоснование |
|--------------|-------------|-------------|
| Уведомление пользователей (GDPR/notification costs) | $15 000 | ~$50 × 300 пострадавших пользователей |
| Форензика и реагирование на инцидент | $20 000 | 4 дня работы специалиста по ИБ |
| Репутационный ущерб (отток клиентов) | $30 000 | 10% отток × средний LTV клиента |
| Регуляторные штрафы (GDPR, при наличии EU-пользователей) | $25 000 | Минимальный штраф для SMB |
| Восстановительные работы (ротация ключей, аудит) | $5 000 | Технические работы |
| **Итого на инцидент (LM)** | **$95 000** | |

**Шаг 3. Ожидаемый годовой ущерб (ALE — Annualized Loss Expectancy)**

```
ALE = TEF × LM = 0,2 × $95 000 = $19 000 в год (для одного проекта)
```

**Шаг 4. Целесообразность мер защиты**

| Мера защиты | Стоимость внедрения | Снижение риска | Экономия ALE |
|-------------|--------------------|--------------------|-------------|
| Secret scanning (GitHub Advanced Security / Gitleaks) | $1 200/год | −70% TEF | $13 300 |
| Автоматическая ротация ключей + аудит | $3 000/год | −20% LM | $3 800 |
| Обучение разработчиков (security awareness) | $500/год | −15% TEF | $2 850 |

> **ROI внедрения secret scanning:** Экономия $13 300 при затратах $1 200 → **ROI ≈ 1 008%**

## 2.5 Сравнительная таблица методов оценки риска

| Критерий | Качественный метод | Полуколичественный (матрица P×L) | FAIR-lite (количественный) |
|----------|--------------------|-----------------------------------|---------------------------|
| **Сложность применения** | Низкая — экспертная оценка | Средняя — шкалы 1–5 | Высокая — требует данных об ущербе |
| **Объективность** | Низкая — субъективные суждения | Средняя — частично субъективная | Высокая — денежное выражение |
| **Сравнимость угроз** | Грубая (4 уровня) | Точная (матрица 5×5 = 25 уровней) | Точная (числовые значения ALE) |
| **Управленческая полезность** | Средняя — понятна руководству | Высокая — приоритизация бюджета | Очень высокая — ROI мер защиты |
| **Скорость применения** | Быстро (часы) | Быстро-средне (дни) | Медленно (недели, сбор данных) |
| **Применимость к данной системе** | Хорошо для первичного скрининга | Оптимально для архитектурных решений | Оптимально для бюджетирования ИБ |
| **Результат для T2** | «Критический» | Риск = 20 (максимум) | ALE = $19 000/год |

### Выводы по сравнению методов:

**Наиболее удобный:** Полуколичественный метод (матрица P×L) — обеспечивает баланс между простотой применения и точностью приоритизации. Позволяет быстро сравнить все 6 угроз и выстроить дорожную карту устранения.

**Наиболее управленчески полезный:** FAIR-lite — единственный метод, дающий ответ на вопрос «сколько стоит внедрение защиты?» и позволяющий рассчитать ROI конкретных мер. Необходим для обоснования бюджета перед руководством.

**Почему нельзя ограничиться одним подходом:**
- Качественный метод быстро даёт общую картину, но не позволяет сравнивать угрозы между собой численно
- Полуколичественный метод помогает расставить приоритеты, но не обосновывает стоимость мер защиты
- FAIR-lite даёт точные числа, но требует данных, которых часто нет на ранних этапах анализа
- Комбинирование трёх методов позволяет: быстро выявить угрозы → приоритизировать их → обосновать инвестиции в защиту

# Выводы

## Наиболее опасные потоки данных

**F1 (E1 → P1, TB1)** и **F5/F12 (E2 → P6, TB1)** — наиболее опасные потоки, поскольку:
- Пересекают внешний периметр (TB1) и принимают данные из недоверенной среды (интернет)
- F1 несёт учётные данные пользователей — компрометация ведёт к ATO-атакам
- F5/F12 несут API-ключи с административными привилегиями — компрометация открывает доступ ко всей пользовательской базе проекта
- Оба потока являются первичными векторами атак: автоматизированные brute-force (F1) и сканеры секретов в репозиториях (F5)

**F8 (E3 → P2, TB3)** — третий по критичности: OAuth callback является сложным протоколом с множеством векторов атак (CSRF, open redirect, code injection), а последствия успешной атаки — полный захват аккаунта.

## Наиболее полезный метод оценки риска

**Полуколичественный метод (матрица P×L)** оказался наиболее практичным для данной системы: позволяет быстро сравнить разнородные угрозы, выстроить приоритеты и обосновать очерёдность работ без сложного сбора финансовых данных. FAIR-lite незаменим для обоснования конкретных бюджетных решений.

## Архитектурные изменения, реально снижающие риск

1. **Единый API Gateway с централизованным rate limiting** для всех внешних эндпоинтов (TB1) — снижает T1 на 70–80%
2. **Хранение API-ключей только в виде хешей** (аналогично паролям) — компрометация D1 не раскрывает ключи
3. **Принудительный PKCE + строгий whitelist redirect_uri** для всех OAuth-потоков — устраняет класс атак на F7/F8
4. **Разделение баз данных** по принципу наименьших привилегий (separate schema per project) — ограничивает blast radius при SQL-инъекции
5. **Event-driven аудит-лог** всех операций над пользователями и разрешениями — обеспечивает детектирование T6 и forensic capability
